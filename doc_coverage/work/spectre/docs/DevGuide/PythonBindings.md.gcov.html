       5             : # Writing Python Bindings {#spectre_writing_python_bindings}
       6             : 
       7             : ## CMake and Directory Layout
       8             : 
       9             : To allow users to analyze output from simulations and take advantage of
      10             : SpECTRE's data structures and functions in python, bindings must sometimes be
      11             : written. SpECTRE uses
      12             : [Boost.Python]
      13             : (https://www.boost.org/doc/libs/1_68_0/libs/python/doc/html/index.html)
      14             : to aid with generating the bindings. The C++ code for the bindings should
      15             : generally go in a `Python` subdirectory. For example, the bindings for the
      16             : DataStructures library would go in `src/DataStructures/Python/`. SpECTRE
      17             : provides the `spectre_python_add_module` CMake function to make adding a new
      18             : python module, be it with or without bindings, easy.  The python bindings are
      19             : built only if `-D BUILD_PYTHON_BINDINGS=ON` is passed when invoking cmake.
      20             : 
      21             : The function `spectre_python_add_module` takes as its first argument the module,
      22             : in our case `DataStructures`. Optionally, a list of `SOURCES` can be passed to
      23             : the CMake function. If you specify `SOURCES`, you must also specify a
      24             : `LIBRARY_NAME`. A good `LIBRARY_NAME` is the name of the C++ library for which
      25             : bindings are being built prefixed with `Py`, e.g. `PyDataStructures`. If the
      26             : Python module will only consist of Python files, then the `SOURCES` option
      27             : should not be specified. Python files that should be part
      28             : of the module can be passed with the keyword `PYTHON_FILES`, e.g.
      29             : `PYTHON_FILES Hello.py HelloWorld.py`. Finally, the `MODULE_PATH` named argument
      30             : can be passed with a string that is the path to where the module should be. For
      31             : example, `MODULE_PATH "submodule0/submodule1/"` would mean the module is
      32             : accessed from python using `import spectre.submodule0.submodule1.MODULE_NAME`.
      33             : 
      34             : Here is a complete example of how to call the `spectre_python_add_module`
      35             : function:
      36             : 
      37             : \code
      38             : spectre_python_add_module(
      39             :   Extra
      40             :   LIBRARY_NAME "PyExtraDataStructures"
      41             :   MODULE_PATH "DataStructures/"
      42             :   SOURCES Bindings.cpp MyCoolDataStructure.cpp
      43             :   PYTHON_FILES CoolPythonDataStructure.py
      44             :   )
      45             : \endcode
      46             : 
      47             : The library that is added has the name `PyExtraDataStructures`. Make sure to
      48             : call `spectre_python_link_libraries` for every Python module that compiles
      49             : `SOURCES`. For example,
      50             : 
      51             : \code
      52             : spectre_python_link_libraries(
      53             :   PyExtraDataStructures
      54             :   PUBLIC ExtraDataStructures
      55             :   )
      56             : \endcode
      57             : 
      58             : You may also call `spectre_python_add_dependencies` for Python modules that
      59             : have `SOURCES`, e.g.
      60             : 
      61             : \code
      62             : spectre_python_add_dependencies(
      63             :   PyExtraDataStructures
      64             :   PyDataStructures
      65             :   )
      66             : \endcode
      67             : 
      68             : Note that these functions will skip adding or configure any C++ libraries if
      69             : the `BUILD_PYTHON_BINDINGS` flag is `OFF`.
      70             : 
      71             : ## Writing Bindings
      72             : 
      73             : Once a python module has been added you can write the actual bindings. You
      74             : should structure your bindings directory to reflect the structure of the library
      75             : you're writing bindings for. For example, say we want bindings for `DataVector`
      76             : and `Matrix` then we should have one source file for each class's bindings
      77             : inside `src/DataStructures/Python`. The functions that generate the bindings
      78             : should be in the `py_bindings` namespace and have a reasonable name such as
      79             : `bind_datavector`. There should be a file named `Bindings.cpp` which calls all
      80             : the `bind_*` functions. The `Bindings.cpp` file is quite simple and should
      81             : `include <boost/python.hpp>`, forward declare the `bind_*` functions, and then
      82             : have `BOOST_PYTHON_MODULE` function. For example,
      83             : 
      84             : \code{.cpp}
      85             : #include <boost/python.hpp>
      86             : 
      87             : namespace py_bindings {
      88             : void bind_datavector();
      89             : }  // namespace py_bindings
      90             : 
      91             : BOOST_PYTHON_MODULE(_PyDataStructures) {
      92             :   Py_Initialize();
      93             :   py_bindings::bind_datavector();
      94             : }
      95             : \endcode
      96             : 
      97             : Note that the library name is passed to `BOOST_PYTHON_MODULE` and is prefixed
      98             : with an underscore. The underscore is important and the library name must be the
      99             : same that is passed as `LIBRARY_NAME` to `spectre_python_add_module` (see
     100             : above).
     101             : 
     102             : The `DataVector` bindings serve as an example with code comments on how to write
     103             : bindings for a class. There is also extensive documentation available directly
     104             : from [Boost.Python]
     105             : (https://www.boost.org/doc/libs/1_68_0/libs/python/doc/html/index.html)
     106             : and [this GitHub repository](https://github.com/TNG/boost-python-examples) also
     107             : has many helpful examples. SpECTRE currently aims to support both Python 2.7
     108             : and Python 3 and as such all bindings must support both.
     109             : 
     110             : \note Exceptions should be allowed to propagate through the bindings so that
     111             : error handling via exceptions is possible from python rather than having the
     112             : python interpreter being killed with a call to `abort`.
     113             : 
     114             : ## Testing Python Bindings and Code
     115             : 
     116             : All the python bindings must be tested. SpECTRE uses the
     117             : [unittest](https://docs.python.org/3/library/unittest.html) framework
     118             : provided as part of python. To register a test file with CMake use the
     119             : SpECTRE-provided function `spectre_add_python_test` passing as the first
     120             : argument the test name (e.g. `"Unit.DataStructures.Python.DataVector"`), the
     121             : file as the second argument (e.g. `Test_DataVector.py`), and a semicolon
     122             : separated list of labels as the last (e.g. `"unit;datastructures;python"`).
     123             : All the test cases should be in a single class so that the python unit testing
     124             : framework will run all test functions on a single invocation to avoid startup
     125             : cost.
     126             : 
     127             : Below is an example of registering a python test file for bindings:
     128             : 
     129             : \snippet tests/Unit/DataStructures/CMakeLists.txt example_add_pybindings_test
     130             : 
     131             : Python code that does not use bindings must also be tested. You can register the
     132             : test file using the `spectre_add_python_test` CMake function with the same
     133             : signature as shown above.
     134             :
